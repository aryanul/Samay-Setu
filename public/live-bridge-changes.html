TL;DR — the four things that changed

If you only read one section, read this one.

  • Trades now have a pillar. The DB has a new column, the API validates it, and the form lets a user pick one. Old trades default to general so nothing breaks.
  • The Live Bridge feed has a search bar and a row of pillar pills (All Trades · Growth & SEO · Finance & Tax · …). Both write to the URL, so filters are bookmarkable and SSR-friendly.
  • The card has a new header badge (PILLAR name on the left, #021 reference on the right), uses OFFERING: / SEEKING: labels per the blueprint, renders the location with a 📍 pin, and ends with a full-width Propose a Bridge CTA.
  • A new script seeds 10 demo members across all pillars so you can see the feed populated immediately. Idempotent — re-running won’t duplicate anyone.

1. A real pillar taxonomy

Until now “pillar” was only a CSS class. The blueprint treats it as a first-class filter dimension, so it needs a stable, lowercase slug list with human labels.

New file: src/lib/pillars.ts

export type PillarSlug =
  | "growth-seo"
  | "finance-tax"
  | "hr-leadership"
  | "prem-lifestyle"
  | "tech-product"
  | "creative-content"
  | "general";

// slug → label, plus helpers
export function pillarLabel(slug): string { … }
export function formatTradeRef(id) { return `#${String(id).padStart(3, "0")}`; }
export function isPillarSlug(value): value is PillarSlug { … }

slug is what lives in the URL and DB. label is what users see on pills and badges. Adding a new pillar means appending one line — nothing else. formatTradeRef(021) produces #021 exactly as the blueprint shows.

2. The database knows about pillars now

A new column, a new index, and a safe backfill — all idempotent.

Changed: the in-process schema bootstrap

ensureSchema() in src/lib/schema.ts now creates and patches the column. 

CREATE TABLE IF NOT EXISTS trades (
  …
  pillar VARCHAR(64) NOT NULL DEFAULT 'general',
  …
  KEY idx_trades_pillar (pillar),
  …
);

// Defensive ALTER for DBs that pre-date this change
ALTER TABLE trades ADD COLUMN pillar VARCHAR(64) NOT NULL DEFAULT 'general';
ALTER TABLE trades ADD KEY idx_trades_pillar (pillar);

The same idempotent ALTER guards live in database/live_bridge.sql so npm run db:live-bridge can fix a stale DB without manual intervention.

Why VARCHAR(64) and not ENUM?

ENUMs are painful to extend on MySQL/TiDB — every new pillar needs a migration. A short VARCHAR with an index and an app-layer allow-list (the isPillarSlug guard) gives the same correctness for free, and adding a pillar is a one-line edit to pillars.ts.

3. The trade form + API accept a pillar

A user can now categorise the trade they’re listing.

API: src/app/api/trades/route.ts

let pillar: PillarSlug = DEFAULT_PILLAR;
if (body.pillar != null && body.pillar !== "") {
  if (!isPillarSlug(body.pillar)) {
    return NextResponse.json({ ok: false, message: "Unknown pillar." }, { status: 422 });
  }
  pillar = body.pillar;
}

await pool.execute(
  `INSERT INTO trades (member_id, skill_offered, skill_needed, location_preference, pillar, status)
   VALUES (?, ?, ?, ?, ?, 'open')`,
  [session.memberId, skillOffered, skillNeeded, locationPreference, pillar]

Why the guard? Anything the client sends is treated as hostile until it matches one of the slugs we know about. Unknown values get a 422 and never reach the DB.

4. The feed page has a search + pill row

URL-driven filtering, debounced search, server-rendered count.

How filtering flows

  1. The user types in the search box or clicks a pill.
  2. The new client component LiveBridgeFilters.tsx rewrites the URL: /dashboard?pillar=finance-tax&q=GST.
  3. Next.js re-renders the server page page.tsx, which reads the params and adds two extra WHERE clauses to the SQL.
  4. The subtitle updates to Active Trades in Finance & Tax (3 Open).

A peek at the resulting toolbar

⌕ Search skills, locations, or keywords…
All Trades Growth & SEO Finance & Tax HR & Leadership Prem Lifestyle Tech & Product Creative & Content

Active Trades in Finance & Tax (4 Open)

Why URL-driven?

  • The server is the source of truth for the visible list — no client/server drift.
  • Refreshing the page keeps the filter. Sharing a URL shares the filter.
  • Search is debounced (280ms) so we don’t pummel the DB on every keystroke.
  • useTransition keeps the UI responsive while the server is fetching — the toolbar fades to opacity 0.65 instead of freezing.

How it stays safe to re-run

  • Each demo member is identified by linkedin_sub = "demo:<slug>". The script checks that key first and reuses the row if it already exists.
  • Each demo trade is matched on (member_id, skill_offered, skill_needed). If it already exists, the script only backfills the pillar — no duplicate rows.
  • To start from scratch, run npm run db:wipe-users (it truncates the member + chat + bridge tables, admin users untouched) and re-seed.
Note Demo accounts do not get login tokens — they exist purely so the Live Bridge feed has cards to render. Log in with your own LinkedIn to actually propose a Bridge to one of them.

How to apply on your machine

Three commands, in this order.

npm run db:live-bridge      # adds trades.pillar + index (idempotent)
npm run db:seed-demo        # 10 demo members across the pillars
npm run dev                 # visit /dashboard

ensureSchema() also self-heals on the first DB hit, so even if you forget db:live-bridge the column will appear when the app boots. The script form exists for ops shells and CI where Node may not start the app.
